Entorno de desarrollo

VSCode + Git / Github

"La mesa de trabajo": donde escribes y versionas el código.

VSCode

  • Ofrece: IntelliSense, terminal, debugger, Git visual, extensiones.
  • Saber: extensiones clave (ESLint, Prettier, Tailwind IntelliSense, GitLens, Supabase), settings.json reproducible, terminal para CLIs.

Frontend

React + Tailwind

"La cara visible". React = estructura/lógica · Tailwind = aspecto. Se anidan, no compiten.

React — lógica de UI

  • Componentes reutilizables, renderizado declarativo, Virtual DOM.
  • Saber: props (datos del padre), estado (useState), useEffect (aquí se llama a Supabase), listas (.map + key), formularios/eventos, custom hooks, y TanStack Query para datos del servidor.

Tailwind — aspecto

  • CSS utility-first, sistema de diseño, responsive, purga del CSS no usado.
  • Saber: tailwind.config.js (tu marca), encapsular patrones repetidos en componentes React, librerías como shadcn/ui.

Infraestructura & hosting

Vercel

"La puerta al mundo": convierte tu código en una web con URL real.

  • Ofrece: despliegue continuo por Git, build automatizado, CDN global, HTTPS/dominios, serverless functions, variables de entorno, preview deployments.
  • Saber: modelo Git (main→producción, otra rama→preview); las funciones serverless para todo lo que no puede vivir en el navegador (email, ops admin, webhooks).
🔒
Variables de entorno: públicas (NEXT_PUBLIC_*) → navegador, solo la anon key. Privadas → solo servidor: aquí viven service_role y la API key de Resend.

Backend & datos

Supabase

El corazón. No es una pieza: es un conjunto de servicios sobre un PostgreSQL real y estándar → conocimiento portable.

  • Database: modelado relacional, FKs, tipos (uuid, timestamptz, numeric para dinero, jsonb), índices, vistas, funciones/triggers, RPC, migraciones con CLI versionadas en Git.
  • Auth: login/OAuth/magic links; emite un JWT con la identidad. Patrón: tabla profiles (FK a auth.users.id) sincronizada por trigger.
  • RLS: reglas de acceso fila a fila usando auth.uid() del JWT. USING (qué ve) vs WITH CHECK (qué escribe).
  • Storage: archivos en buckets, también con RLS, signed URLs para acceso temporal.
  • Realtime: cambios en vivo (WebSockets), respeta RLS.
  • PostgREST: API REST automática desde tu esquema; supabase-js lleva el JWT y pasa por RLS.
  • Edge Functions: backend propio (Deno/TS). Se solapan con Vercel → para el MVP, una sola plataforma de funciones.
⚠
RLS es lo más importante. La anon key del frontend es pública → la seguridad no puede vivir en React. Vive en la base de datos. RLS activado sin políticas = nadie accede.
🔒
La service_role se salta toda la RLS → es la llave maestra, solo en servidor.

Comunicaciones

Resend

"La voz hacia el exterior": email transaccional (facturas, confirmaciones, recordatorios).

  • Ofrece: API de email transaccional, React Email (plantillas como componentes), gestión de dominios, webhooks de eventos, logs.
⚠
API key = secreto absoluto. Solo en una serverless function, jamás en el navegador.
⚠
Entregabilidad: verificar dominio con SPF (quién envía), DKIM (firma anti-manipulación) y DMARC (qué hacer si fallan). Servicio facturable en sí mismo.

Parte 3

Flujo integrado · caso real

Un cliente solicita cita desde la web → consulta gratuita → contrata y firma (datos de su hijo) → la administradora le da de alta y programa la agenda. Recorre los 5 departamentos y 3 actores.

  • Fase 1 · Captación — formulario de cita (3·2·4·5)
  • Fase 2 · Gestión + agenda — profesional lee solicitudes (RLS) y asigna hueco (2·4·5)
  • Fase 3 · Conversión — tarifa, datos del hijo, firma; doc a Storage privado (2·3·4)
  • Fase 4 · Alta formal — administradora da de alta y programa sesiones recurrentes (2·3·4·5)

El patrón que se repite

patrón base
HUMANO ──▶ WEB(React) ──▶ FN(Vercel) ──▶ { SUPABASE  +  RESEND }
(actúa)    (recoge)        (orquesta,      (guarda /    (comunica
                          guarda secretos)  consulta)    al exterior)
  1. El frontend nunca actúa solo: quien cruza hacia datos/servicios es siempre la función de Vercel (custodia los secretos).
  2. Supabase es el punto de verdad compartido: todos los actores leen/escriben en la misma DB; RLS decide qué ve cada rol.
  3. Resend es la última milla: al final de cada fase, disparado por la función, nunca por el navegador.
